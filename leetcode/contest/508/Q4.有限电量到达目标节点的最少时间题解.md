## 1. 题目概述

- **题目名称**：Q4. 有限电量到达目标节点的最少时间
- **链接**：[3977. 有限电量到达目标节点的最少时间](https://leetcode.cn/problems/minimum-time-to-reach-target-with-limited-power/)
- **来源**：LeetCode 第 508 场周赛 Q4
- **难度**：困难
- **标签**：图、最短路、Dijkstra

**题意简述**：给定一个有向带权图，每个节点 `u` 有一个转发耗电 `cost[u]`。信号从 `source` 出发，初始电量为 `power`，要到达 `target`。只有当剩余电量不少于 `cost[u]` 时，才能从节点 `u` 出发沿某条出边前进，经过边会消耗时间 `ti`，经过节点会消耗电量 `cost[u]`。求到达 `target` 的最小时间；若存在多条最短时间路径，返回其中剩余电量最大者对应的剩余电量。

**返回值**：`[最小时间, 最大剩余电量]`，无法到达则返回 `[-1, -1]`。

**关键约束**：

- `1 <= n <= 1000`
- `0 <= edges.length <= 1000`
- `1 <= ti <= 10^9`
- `1 <= power <= 1000`
- `1 <= cost[i] <= 2000`

## 2. 示例

### 示例 1

```text
n = 5
edges = [[0,1,1],[1,4,1],[0,2,1],[2,3,1],[3,4,1]]
power = 4
cost = [2,3,1,1,1]
source = 0, target = 4
```

- 路径 `0 -> 1 -> 4`：离开 0 后剩余 `4 - 2 = 2 < cost[1] = 3`，不合法。
- 路径 `0 -> 2 -> 3 -> 4`：时间 `1 + 1 + 1 = 3`，耗电 `cost[0] + cost[2] + cost[3] = 2 + 1 + 1 = 4`，剩余 `0`。

输出：`[3, 0]`

### 示例 2

```text

power = 3
cost = [1,1,1]
source = 1, target = 1
```

起点即终点，无需移动。

输出：`[0, 3]`

### 示例 3

```text
n = 4
edges = [[0,1,3],[2,3,4]]
power = 3
cost = [1,1,1,1]
source = 0, target = 3
```

不存在从 0 到 3 的路径。

输出：`[-1, -1]`

## 3. 解题思路

### 3.1 暴力思路

枚举所有从 `source` 到 `target` 的简单路径，计算每条路径的时间和耗电，筛选合法路径后取时间最小、剩余电量最大者。

- 时间复杂度：最坏 `O(n!)`，不可接受。
- 空间复杂度：`O(n)`。

### 3.2 反例分析：单状态 Dijkstra 为何不够

若用普通 Dijkstra：每个节点保留 `(最短时间, 该时间下的最小耗电)`，这个做法**不正确**。

**反例**：

```text
节点：0, 1, 2, 3
边：0→3(t=1), 3→1(t=1), 0→1(t=3), 1→2(t=1)
cost = [1, 1, 1, 10], power = 11, source = 0, target = 2
```

- 路径 `0→3→1`：到达 1 的时间为 2，但已耗电 1+10=11，剩余 0，无法再走 `1→2`。
- 路径 `0→1`：到达 1 的时间为 3，已耗电 1，剩余 10，可以走 `1→2`。

如果只保留“到达 1 的最短时间状态”`(2, 11)`，就会误判 1 是死路，漏掉正确答案 `[4, 9]`。

![反例图](images/min_time_max_power_counterexample.png)

**核心原因**：时间更短的路径可能耗电更多，导致后续因电量不足无法继续；而一条稍慢但更省电的路径反而能走通。因此每个节点不能只保留一个状态。

### 3.3 正确思路：按“耗电”拆点的 Dijkstra

由于 `power <= 1000`，可以把**已耗电量**也作为状态的一维：

```text
dp[u][p] = 从 source 到达节点 u，且总耗电恰好为 p 时的最小时间
```

状态转移：在节点 `u` 且已耗电 `p` 时，若 `p + cost[u] <= power`，可以走边 `u→v`（耗时 `t`），转移到 `dp[v][p + cost[u]]`：

```text
dp[v][p + cost[u]] = min(dp[v][p + cost[u]], dp[u][p] + t)
```

最终答案：

- `minTime = min_p dp[target][p]`
- 在所有满足 `dp[target][p] == minTime` 的 `p` 中取最小值，剩余电量 = `power - minPower`
- 若 `minTime == INF`，返回 `[-1, -1]`

这等价于把原图的每个节点 `u` 拆成 `power + 1` 个状态点 `(u, p)`，然后在新图上跑 Dijkstra。

## 4. 算法细节

1. 建图：使用邻接表存储有向边。
2. 初始化：`dp[source][0] = 0`，其余 `dp[u][p] = INF`。
3. 优先队列：元素为 `(time, power_consumed, node)`，按时间优先排序。
4. 每次取出队首状态 `(t, p, u)`：
   - 若 `t > dp[u][p]`，说明该状态已被更新过，跳过。
   - 遍历 `u` 的每条出边 `u→v`（耗时 `w`）：
     - 若 `p + cost[u] > power`，电量不足，跳过。
     - 更新 `dp[v][p + cost[u]] = min(dp[v][p + cost[u]], t + w)`，并入队。
5. 最终从 `dp[target][p]`（`0 <= p <= power`）中找出最小时间，以及对应的最小耗电，计算剩余电量。

## 5. 正确性证明

**引理 1**：对于任意状态 `(u, p)`，当算法从优先队列中第一次取出满足 `curTime == dp[u][p]` 的状态时，`dp[u][p]` 已经是从 `source` 到达 `(u, p)` 的最小时间。

**证明**：新图的所有边权（即原图边的时间 `t`）均为正数。优先队列按时间从小到大弹出状态，因此第一次从队列中取出 `(u, p)` 时，其时间一定是所有到达 `(u, p)` 的路径中最小的。后续的松弛不会再产生更小的时间。∎

**引理 2**：如果存在一条从 `source` 到 `target` 的合法路径，总耗时为 `T`，总耗电为 `P`，那么算法最终会有 `dp[target][P] <= T`。

**证明**：对路径上的节点和边进行归纳。初始 `dp[source][0] = 0`。假设路径上经过节点 `u` 时 `dp[u][p] <=` 路径到 `u` 的实际时间。从 `u` 出发走合法边到 `v`，耗电增加 `cost[u]`，时间增加 `t`，算法在松弛时会尝试更新 `dp[v][p + cost[u]]`。因此最终 `dp[target][P]` 不会大于路径实际时间 `T`。∎

**定理**：算法返回的 `answer` 满足题意。

**证明**：

- 若 `target` 不可达，由引理 2 可知所有 `dp[target][p]` 保持 INF，`minTime == INF`，返回 `[-1, -1]` 正确。
- 若可达，设实际最小时间为 `T*`。由引理 2，存在某个 `p` 使得 `dp[target][p] <= T*`；而 `T*` 是下界，故 `minTime = T*`。
- 在所有 `dp[target][p] == T*` 的状态中，我们取最小耗电 `P*`，则剩余电量 `power - P*` 是所有最短时间路径中的最大剩余电量。∎

## 6. 复杂度分析

- **状态数**：`n * (power + 1)`，最多 `1000 * 1001 ≈ 10^6`。
- **时间复杂度**：`O((n * power + m * power) * log(n * power))`。每个状态最多入队一次，每次优先队列操作 `O(log(n * power))`；每条原图边在每种耗电状态下都可能触发一次松弛，共 `O(m * power)` 次松弛。
- **空间复杂度**：`O(n * power + m)`，用于 `dp` 数组和邻接表。

## 7. 参考代码

### C++

```cpp
class Solution {
  public:
    vector<long long> minTimeMaxPower(int n, vector<vector<int>>& edges, int power, vector<int>& cost, int source,
                                      int target) {
        using State = tuple<long long, int, int>; // (time, power_consumed, node)
        // 注意：力扣在序列化 long long 时要求值落在 [-(2^53-1), 2^53-1] 范围内。
        // 因此 INF 必须小于 2^53。1e15 足够大（大于任何合法时间 1e12），且安全。
        const long long INF = (long long)1e15;

        // 建图
        vector<vector<pair<int, long long>>> g(n);
        for (auto& e : edges) {
            int u = e[0], v = e[1];
            long long t = e[2];
            g[u].push_back({v, t});
        }

        // dp[u][p] = 到达 u 且总耗电为 p 时的最小时间
        vector<vector<long long>> dp(n, vector<long long>(power + 1, INF));
        dp[source][0] = 0;

        // 优先队列：按时间优先
        priority_queue<State, vector<State>, greater<State>> pq;
        pq.push({0, 0, source});

        while (!pq.empty()) {
            auto [curTime, curPower, u] = pq.top();
            pq.pop();

            // 跳过已被更优状态覆盖的过时状态
            if (curTime > dp[u][curPower])
                continue;

            for (auto& [v, t] : g[u]) {
                // 检查是否还有足够电量从 u 转发
                int np = curPower + cost[u];
                if (np > power)
                    continue;

                long long nt = curTime + t;
                if (nt < dp[v][np]) {
                    dp[v][np] = nt;
                    pq.push({nt, np, v});
                }
            }
        }

        // 找最短时间
        long long minTime = INF;
        for (int p = 0; p <= power; p++) {
            minTime = min(minTime, dp[target][p]);
        }

        if (minTime == INF) {
            return {-1, -1};
        }

        // 在所有最短时间中找耗电最小的，从而剩余电量最大
        long long minPower = INF;
        for (int p = 0; p <= power; p++) {
            if (dp[target][p] == minTime) {
                minPower = min(minPower, (long long)p);
            }
        }

        return {minTime, power - minPower};
    }
};
```

### Python

```python
import heapq
from typing import List

class Solution:
    def minTimeMaxPower(self, n: int, edges: List[List[int]], power: int,
                        cost: List[int], source: int, target: int) -> List[int]:
        INF = float('inf')
        g = [[] for _ in range(n)]
        for u, v, t in edges:
            g[u].append((v, t))

        # dp[u][p] = 到达 u 且总耗电为 p 时的最小时间
        dp = [[INF] * (power + 1) for _ in range(n)]
        dp[source][0] = 0

        pq = [(0, 0, source)]  # (time, power_consumed, node)
        while pq:
            curTime, curPower, u = heapq.heappop(pq)
            if curTime > dp[u][curPower]:
                continue
            np = curPower + cost[u]
            if np > power:
                continue
            for v, t in g[u]:
                nt = curTime + t
                if nt < dp[v][np]:
                    dp[v][np] = nt
                    heapq.heappush(pq, (nt, np, v))

        minTime = min(dp[target])
        if minTime == INF:
            return [-1, -1]
        minPower = min(p for p in range(power + 1) if dp[target][p] == minTime)
        return [minTime, power - minPower]
```

## 8. 边界情况与易错点

| 情况 | 说明 |
|------|------|
| `source == target` | 直接返回 `[0, power]`，无需遍历。 |
| 无出边 / 不可达 | 返回 `[-1, -1]`。 |
| 电量不足 | 即使时间更短，若 `curPower + cost[u] > power` 也不能走这条边。 |
| 一状态 Dijkstra 错误 | 每个节点只保留 `(最短时间, 最小耗电)` 会漏解：更快路径可能耗电更多导致后续断电，更慢路径反而能走通。 |
| 状态数 | `dp` 数组大小为 `n * (power + 1)`，注意 `power <= 1000`，不会爆内存。 |
| 大数溢出 | `ti` 最大 `10^9`，路径最多 `n` 条边，时间最大约 `10^12`，需用 `long long`。 |
| INF 取值过大 | 若用 `LLONG_MAX / 4` 作 INF，力扣在序列化 `long long` 时可能因超出 JSON 安全整数范围 `[-(2^53-1), 2^53-1]` 而报错。建议使用 `1e15` 左右的 INF，并显式构造返回数组。 |
